# Keyspace region-label index scalability

Canonical keyspace label rules are indexed separately from generic region-label
rules. A keyspace rule is recognized only when its ID, label, and RawKV/Txn
boundaries exactly match the shape generated by `pkg/keyspace`. Canonical rules
use sparse fixed-size slots and boundary bitsets; generic rules continue to use
the range list.

This keeps a single canonical-rule insertion or deletion bounded: it does not
traverse or sort existing rules, and it does not build a replacement range
index while holding the labeler lock. RawKV-disabled rules occupy one slot and
RawKV-enabled rules occupy one slot in each mode.

## Readiness target

After rules have been delivered to the process, building the in-memory index
for 1,000,000 canonical Txn rules should finish within 5 seconds on the
reference development environment. Storage paging and etcd latency are outside
this in-memory target and should be measured separately when evaluating
end-to-end Scheduling Service readiness.

The index has bounded structural overhead. Per populated API mode it allocates
a 2 MiB presence bitset, a 128 KiB chunk directory, and one 8 KiB slot chunk
per occupied 1,024-ID block (128 KiB per 16,384 densely populated keyspaces).
Rule objects remain owned by the main rule map rather than being duplicated by
the index.

## Scale benchmark

Run the checked-in benchmarks at both required sizes:

```sh
go test ./pkg/schedule/labeler -run '^$' \
  -bench '^BenchmarkRegionLabelerKeyspaceIndex/startup/(100000|1000000)$' \
  -benchtime=1x -benchmem
go test ./pkg/schedule/labeler -run '^$' \
  -bench '^BenchmarkRegionLabelerKeyspaceIndex/single-rule-update/(100000|1000000)$' \
  -benchtime=100000x -benchmem
```

For peak RSS on macOS, compile once so compiler memory is not included, then
measure each startup size independently:

```sh
go test -c -o /tmp/pd-labeler-bench.test ./pkg/schedule/labeler
/usr/bin/time -l /tmp/pd-labeler-bench.test -test.run '^$' \
  -test.bench '^BenchmarkRegionLabelerKeyspaceIndex/startup/100000$' \
  -test.benchtime=1x -test.benchmem
/usr/bin/time -l /tmp/pd-labeler-bench.test -test.run '^$' \
  -test.bench '^BenchmarkRegionLabelerKeyspaceIndex/startup/1000000$' \
  -test.benchtime=1x -test.benchmem
```

Reference results on `darwin/amd64` (`VirtualApple @ 2.50GHz`) are:

| Operation | Rules | Time | Allocated bytes | Allocations | Peak RSS |
| --- | ---: | ---: | ---: | ---: | ---: |
| Startup | 100,000 | 144 ms | 95,376,336 | 1,999,889 | 89,001,984 bytes |
| Startup | 1,000,000 | 1.70 s | 955,328,440 | 20,004,601 | 668,168,192 bytes |
| Single-rule update | 100,000 | 156 ns | 0 | 0 | unchanged steady state |
| Single-rule update | 1,000,000 | 154 ns | 0 | 0 | unchanged steady state |

The single-rule benchmark holds the same exclusive labeler lock used by the
production in-memory update, so its duration is also the measured lock hold
time for this path.
